# Portfolio UX Audit — Full Report

## Executive Summary

The audit covers 5 sections (Hero, About, Experience, Projects, Contact) and surfaced **64 findings**: 18 critical, 31 warnings, and 15 suggestions. The single most pervasive cross-cutting defect is the complete absence of `prefers-reduced-motion` guards — it affects all 5 sections unconditionally, violating WCAG 2.1 SC 2.3.3 and risking harm to vestibular-sensitive users. Color contrast failures on `--color-text-muted` appear in 4 sections (Hero, About, Experience, Contact) and GSAP `opacity:0`-with-no-CSS-fallback leaves entire sections blank on JS failure in 3 sections (Hero, Projects, Contact). Overall UX health is visually strong but the site is not production-ready for accessibility compliance without addressing the critical tier. **Top 3 priorities**: (1) add a global `prefers-reduced-motion` guard across all GSAP timelines and CSS keyframe animations; (2) lighten `--color-text-muted` to meet the WCAG AA 4.5:1 threshold on all backgrounds; (3) add CSS `opacity:1` fallbacks for every element that GSAP initialises at `opacity:0`.

## Cross-Cutting Patterns

### 1. Missing `prefers-reduced-motion` guard — affects **all 5 sections**
- **Sections**: Hero, About, Experience, Projects, Contact
- **Where**: `hero.space.animations.js:10–44`, `About.jsx:12–28`, `Experience.jsx:103–119`, `Projects.jsx:11-23 & 141-153`, `Contact.jsx:38–49 & 64–75`
- **Impact**: WCAG 2.1 SC 2.3.3 violation; can trigger vestibular disorders for users with vestibular motion sensitivity.
- **Single best fix**: Add one global `gsap.matchMedia()` wrapper in each section's animation `useEffect`. Additionally add the CSS safety-net in `index.css`: `@media (prefers-reduced-motion: reduce) { *, *::before, *::after { animation-duration: 0.01ms !important; transition-duration: 0.01ms !important; } }`. This catches both GSAP-driven and CSS keyframe animations in one declaration.

### 2. `--color-text-muted` contrast failure — affects **4 sections** (Hero, About, Experience, Contact)
- **Sections**: Hero (critical, ≈4.28:1), About (critical, ≈4.1:1), Experience (suggestion, contextual risk at `text-xs`), Contact (warning, ≈4.39:1)
- **Where**: `index.css:9` (token definition), consumed in `Hero.jsx:111`, `About.jsx:65`, `Experience.jsx:54`, `Contact.jsx:101`
- **Impact**: WCAG AA failure for normal-weight body text at 16 px on all dark backgrounds.
- **Single best fix**: Change the single token definition in `index.css` to `--color-text-muted: #8899b0` (≈6:1 on `#050508`, ≈5.8:1 on `#0d0d14`). This one-line fix resolves the contrast violation in all four sections simultaneously.

### 3. GSAP `opacity:0` with no CSS fallback — affects **3 sections** (Hero, Projects, Contact)
- **Sections**: Hero, Projects, Contact
- **Where**: `hero.space.animations.js:21–23`, `Projects.jsx:142`, `Contact.jsx:38–49`
- **Impact**: Entire section content goes permanently invisible when JS fails, is blocked, or GSAP fails to load.
- **Single best fix**: Establish a CSS convention — never set initial visibility via GSAP `from()`; instead write a `.js-loading` class on `<body>` via JS immediately at page load and apply `opacity: 0` only inside that scope: `.js-loading .hero-fade, .js-loading .project-card, .js-loading .contact-animate { opacity: 0; }`. Remove the class after GSAP initialises. No-JS and GSAP-fail users will always see content.

### 4. Decorative section numbers not `aria-hidden` — affects **3 sections** (About, Experience, Contact)
- **Sections**: About (`01.`), Experience (referenced in About finding as `Experience.jsx:131–135`), Contact (`04.`)
- **Where**: `About.jsx:39–44`, `Experience.jsx:131–135`, `Contact.jsx:83`
- **Impact**: Screen readers announce "zero one dot", "zero four dot", etc. as part of heading content, polluting the accessible name and confusing landmark navigation.
- **Single best fix**: Add `aria-hidden="true"` to every section-number `<span>` across the codebase. A global grep for `"01."`, `"02."`, `"03."`, `"04."` in JSX will find all instances.

### 5. Hardcoded `rgba`/hex magic values bypassing CSS tokens — affects **all 5 sections**
- **Sections**: Hero (`Hero.jsx:129–134`), About (`About.jsx:52–53`), Experience (`Experience.jsx:31,33`), Projects (`Projects.jsx:47,122–124`), Contact (`Contact.jsx:65,117–126`)
- **Impact**: Design token changes to `--color-accent-cyan`, `--color-accent-purple`, or `--color-bg-secondary` will silently not propagate to hardcoded values, causing visual drift.
- **Single best fix**: Extend `index.css` `@theme` block with pre-computed opacity variants (e.g., `--color-accent-cyan-15`, `--color-accent-cyan-80`) and replace all `rgba(0,245,255,…)` instances with `color-mix(in srgb, var(--color-accent-cyan) <N>%, transparent)`. A codebase-wide find-replace for `rgba(0,245,255` and `rgba(139,92,246` will locate all occurrences.

### 6. Duplicate Tailwind utilities + overriding inline styles — affects **3 sections** (About, Experience, Projects)
- **Sections**: About (`About.jsx:34–35,37`), Experience (`Experience.jsx:125–127`), Projects (`Projects.jsx:116,187–188`)
- **Impact**: Tailwind responsive modifiers (`md:py-32`, `sm:flex-row`, etc.) are silently defeated by inline style wins, causing dead utility classes and preventing mobile-first responsive overrides.
- **Single best fix**: Establish a team rule — never mix inline `style={{}}` and Tailwind utilities for the same CSS property on the same element. Adopt Tailwind utilities as the single source of truth for layout and spacing; remove all inline `style` padding/gap/flex-direction overrides.
