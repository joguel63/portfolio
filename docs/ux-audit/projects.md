# UX Audit — Projects

## Summary

The Projects section is visually strong and architecturally sound, but has two critical accessibility gaps (non-descriptive link labels that produce duplicate "GitHub / GitHub / GitHub" announcements for screen readers) and one critical progressive-enhancement failure (cards are invisible if GSAP fails to initialise). The 3D tilt and scroll-entrance animations also lack `prefers-reduced-motion` guards, which is a policy-level violation per the project's own pre-delivery checklist.

## Findings

| # | Severity | Dimension | Finding | File:Line | Suggested Fix |
|---|----------|-----------|---------|-----------|---------------|
| 1 | 🔴 Critical | Accessibility | `aria-label="GitHub"` on every card's GitHub link is non-descriptive. A screen-reader user tabbing through all cards hears "GitHub, GitHub, GitHub…" with no way to distinguish projects. | `Projects.jsx:77` | Change to `aria-label={\`GitHub repository for ${project.title}\`}` |
| 2 | 🔴 Critical | Accessibility | `aria-label="Live demo"` on every card's live link has the same problem — all five (or three) links are announced identically, giving no project context. | `Projects.jsx:90` | Change to `aria-label={\`Live demo for ${project.title}\`}` |
| 3 | 🔴 Critical | Perceived Performance | `gsap.from('.project-card', { opacity: 0, … })` sets initial opacity to 0 via JavaScript. If GSAP fails to load (network error, JS disabled), all project cards remain permanently invisible — there is no CSS fallback. | `Projects.jsx:142` | Add `.project-card { opacity: 1 }` as a CSS baseline; GSAP's `gsap.set` or a `noscript` rule ensures graceful degradation. |
| 4 | 🔴 Critical | Perceived Performance | Neither the scroll-entrance animation nor the mousemove tilt effect checks `prefers-reduced-motion`. The project's own pre-delivery checklist (ui-ux-pro-max skill) and WCAG 2.1 §2.3.3 require this. | `Projects.jsx:11-23, 141-153` | Wrap both animations: `const reducedMotion = window.matchMedia('(prefers-reduced-motion: reduce)').matches`; skip or snap-to-final-state when true. |
| 5 | 🟡 Warning | Accessibility | Project cards are plain `<div>` elements with no semantic grouping role. Screen readers cannot identify each card as a self-contained content unit. | `Projects.jsx:41` | Replace the outer `<div>` with `<article>` (or add `role="article"`) and optionally `aria-label={project.title}`. |
| 6 | 🟡 Warning | Accessibility | The 3D card tilt is triggered only by `mousemove` — keyboard and touch users never see it. While purely decorative, `onFocus` could fire a subtle CSS-based lift to give keyboard users equivalent tactile feedback without requiring motion. | `Projects.jsx:43-44` | Add an `onFocus` handler that applies a static `translateY(-4px)` transform (skipping the 3D rotation) so keyboard focus still provides a hover-equivalent visual cue. |
| 7 | 🟡 Warning | Perceived Performance | `handleMouseMove` calls `gsap.to()` on every `mousemove` event — potentially 60–100 calls per second per card. Even with `overwrite: 'auto'`, each call incurs GSAP's tween lookup cost. There is no `requestAnimationFrame` gate or pointer-coordinates delta threshold. | `Projects.jsx:11-24` | Guard with a `rafId` ref: cancel the previous frame and schedule a new one, so only one `gsap.to()` fires per animation frame. |
| 8 | 🟡 Warning | Visual Consistency | `cursor: 'default'` is explicitly set on cards that produce a neon-glow border, box-shadow, and 3D tilt on hover. This is the opposite of the signal the design sends — users expect `cursor: pointer` on visually interactive surfaces. The links inside also inherit default cursor. | `Projects.jsx:51` | Change to `cursor: 'pointer'`, consistent with the ui-ux-pro-max checklist rule *"cursor-pointer on all clickable/hoverable cards"*. |
| 9 | 🟡 Warning | Accessibility | When `project.live` is `null` (projects 2, 4, 5 in `projects.js`), the live link silently disappears. There is no affordance — visual or accessible — telling users this project has no deployed demo, which can read as a broken or loading state. | `Projects.jsx:85-98` / `projects.js:18,34,43` | Consider a visually muted, non-interactive `<span aria-label="No live demo available">Live</span>` or a tooltip/title attribute on the absent link's container. |
| 10 | 🟡 Warning | Responsiveness | `transformStyle: 'preserve-3d'` forces a GPU compositing layer for every card. On low-memory mobile devices (especially iOS Safari with five simultaneous composited layers), this can trigger layer eviction and re-rasterisation, causing visible jank on scroll. | `Projects.jsx:49` | Apply `preserve-3d` only when the pointer is inside the card (`onMouseEnter`/`onMouseLeave`) and reset to `flat` on leave, limiting compositing to the active card. |
| 11 | 🟢 Suggestion | Responsiveness | `className="grid gap-6"` and `style={{ gap: '1.5rem' }}` both set the same 1.5 rem gap. The inline style wins in the CSS cascade but the Tailwind class is redundant and misleading for maintainers. The same pattern repeats with `flex flex-wrap gap-2` + `style={{ display:'flex', flexWrap:'wrap', gap:'0.5rem' }}` on line 116. | `Projects.jsx:187-188, 116` | Remove duplicated Tailwind utility classes when an inline style already owns that property, or vice-versa. Pick one canonical source. |
| 12 | 🟢 Suggestion | Visual Consistency | `backgroundColor: 'rgba(13,13,20,0.8)'` hard-codes the RGB components of `--color-bg-secondary: #0d0d14` (#0d0d14 = rgb(13,13,20)). If the design token changes, the card colour will silently diverge. | `Projects.jsx:47` / `index.css:5` | Use `color-mix(in srgb, var(--color-bg-secondary) 80%, transparent)` or a dedicated `--color-card-bg` token in `index.css`. |
| 13 | 🟢 Suggestion | Visual Consistency | Tech-tag styles (`rgba(139,92,246,0.15)` background and `rgba(139,92,246,0.3)` border) hard-code the RGB of `--color-accent-purple` (#8b5cf6 = rgb(139,92,246)). Same risk as finding #12 and inconsistent with how other sections may reference this token. | `Projects.jsx:122-124` | Replace with `color-mix(in srgb, var(--color-accent-purple) 15%, transparent)` for the background and `color-mix(in srgb, var(--color-accent-purple) 30%, transparent)` for the border. |
| 14 | 🟢 Suggestion | Visual Consistency | The `↗` arrow glyph in the visible link text ("GitHub ↗", "Live ↗") is overridden by the `aria-label` attribute, so it is correctly hidden from screen readers as-is. However, it is not explicitly `aria-hidden`, meaning if the `aria-label` were ever removed, the arrow would be announced as "north east arrow" by VoiceOver/NVDA. Making intent explicit is a resilience improvement. | `Projects.jsx:83, 96` | Wrap the arrow: `<span aria-hidden="true"> ↗</span>` so the decoration is permanently screen-reader-silent regardless of the outer label. |
