# UX Audit — Hero

## Summary

The Hero section delivers a visually striking cyberpunk aesthetic with fluid typography and a polished space animation, but carries several critical accessibility defects: the CTA link has no keyboard focus style, the decorative WebGL canvas is not hidden from screen readers, muted text (`#64748b` on `#050508`) fails WCAG AA contrast at ≈4.28:1, and the entire section goes invisible if JavaScript fails or `prefers-reduced-motion` is not respected. Responsiveness and perceived performance also have notable gaps — `h-screen` breaks on iOS Safari with dynamic toolbars, and 5 000 GL stars plus per-character `filter: blur()` can cause jank on low-end devices.

## Findings

| # | Severity | Dimension | Finding | File:Line | Suggested Fix |
|---|----------|-----------|---------|-----------|---------------|
| 1 | 🔴 Critical | Accessibility | CTA `<a href="#projects">` has **no `:focus-visible` style** — only JS `onMouseEnter`/`onMouseLeave` handlers manage visual state. Keyboard users get no focus indicator. | `Hero.jsx:116–135`, `index.css:61–67` | Add to `index.css`: `a:focus-visible { outline: 2px solid var(--color-accent-cyan); outline-offset: 4px; }`. The existing `a:hover` rule offers no help for keyboard navigation. |
| 2 | 🔴 Critical | Accessibility | `<Canvas>` (Three.js/WebGL) is a **decorative scene** but has no `aria-hidden="true"` — screen readers will attempt to traverse its DOM subtree and may announce spurious content or crash AT. | `Hero.jsx:73–82` | Add `aria-hidden="true"` as a prop to `<Canvas>`: `<Canvas aria-hidden="true" …>`. |
| 3 | 🔴 Critical | Accessibility | **No `prefers-reduced-motion` check** anywhere — the continuous star rotation (`useFrame`), character blur-in, and entrance timeline run unconditionally, violating WCAG 2.3.3 (Animation from Interactions). | `hero.space.animations.js:10–44`, `Hero.jsx:10–15` | Guard the GSAP timeline: `if (window.matchMedia('(prefers-reduced-motion: reduce)').matches) { gsap.set([...chars,...fades,canvas],{opacity:1,y:0,filter:'none'}); return null; }`. Also stop `useFrame` rotation when reduced-motion is preferred. |
| 4 | 🔴 Critical | Accessibility | **Color contrast failure**: `--color-text-muted: #64748b` on `--color-bg-primary: #050508` yields ≈**4.28:1** — below the WCAG AA threshold of 4.5:1 for normal-weight body text. The tagline paragraph (`text-base`, 16 px, regular weight) uses this token directly. | `index.css:4,9`, `Hero.jsx:111` | Lighten `--color-text-muted` to `#7b91a8` (≈4.6:1) or restrict this token to large/bold text only (≥18 px or bold ≥14 px, which requires only 3:1). |
| 5 | 🔴 Critical | Perf | **JS-only content visibility**: GSAP sets `opacity: 0` on all `.hero-char`, `.hero-fade`, and `canvas` elements before animation. If JS fails or is blocked, the entire Hero is blank — no fallback text, no `<noscript>` block. | `hero.space.animations.js:21–23` | Apply `opacity: 0` via a `.js-hero-init` CSS class added by JS at runtime. If JS never runs, elements remain visible. Add `<noscript>` styles to `index.css` as a safety net. |
| 6 | 🟡 Warning | Accessibility | **Scroll indicator not hidden from AT**: The bottom `<div>` with "scroll" text and the animated cyan line is purely decorative but has no `aria-hidden="true"`. Screen readers will announce "scroll" as a word in the middle of page content flow. | `Hero.jsx:141–150` | Add `aria-hidden="true"` to the outer scroll indicator `<div>` (line 141). |
| 7 | 🟡 Warning | Accessibility | **Incomplete `<h1>` label**: The `<h1>` contains only `profile.name` ("Miguel Muñoz"). The greeting prefix "Hola, soy" is in a sibling `<p>`, so AT heading navigation announces the heading without context. Not technically wrong but creates a disjointed reading experience. | `Hero.jsx:85–97` | Add `aria-label="Hola, soy Miguel Muñoz"` to `<h1>` so assistive technologies announce the full intended phrase, while the visual split layout is preserved. |
| 8 | 🟡 Warning | Responsiveness | **`h-screen` (100vh) breaks on iOS Safari**: The dynamic address/tab bar on iOS reduces the actual viewport, causing the hero section to overflow below the visible area — a well-known bug affecting all iOS browsers. | `Hero.jsx:69` | Replace `h-screen` with `h-[100dvh]` (Tailwind JIT, supported in Safari 15.4+). Add `h-screen` as a fallback for older browsers: `class="h-screen h-[100dvh]"`. |
| 9 | 🟡 Warning | Responsiveness | **Name overflow at 375 px**: `clamp(3rem, 8vw, 7rem)` resolves to **48 px** at 375 px viewport. "Miguel Muñoz" (12 chars, bold Space Grotesk) renders ≈336 px wide inside a `px-6` container (~327 px available) — likely forcing an awkward mid-word wrap. | `Hero.jsx:69,94` | Lower the clamp minimum to `2.5rem`: `clamp(2.5rem, 8vw, 7rem)`. Alternatively add `whitespace-nowrap` and let the clamp handle scaling, verifying at 320 px, 375 px, and 390 px. |
| 10 | 🟡 Warning | Perf | **`<Stars count={5000}>` is GPU-heavy on low-end mobile**: 5 000 instanced GL points rendered every frame, combined with scroll-driven rotation, can cause thermal throttling and dropped frames on budget Android devices. | `Hero.jsx:19–26` | Reduce count based on device capability: detect via `navigator.hardwareConcurrency < 4` or `window.devicePixelRatio < 2` and use `count={2000}` on constrained devices. `@react-three/drei`'s `Detect` utility can also help. |
| 11 | 🟡 Warning | Perf | **Per-character `filter: blur()` creates many paint layers**: GSAP animates `filter: blur(10px)` on each of the 12 individual `.hero-char` `<span>` elements. Each blurred element forces the browser to composite it on a separate layer, multiplying paint cost — especially on mobile GPUs. | `hero.space.animations.js:21,35` | Apply blur animation to the parent `<span>` wrapping the whole name (in `SplitText`) rather than per-character, or add `will-change: filter` to `.hero-char` in CSS to pre-promote layers before animation starts. |
| 12 | 🟡 Warning | Perf | **No `ErrorBoundary` around `<Canvas>`**: If WebGL context creation fails (common in low-memory tabs or privacy browsers), React throws an unhandled error that collapses the entire Hero section — leaving a blank space with no fallback. | `Hero.jsx:73–82` | Wrap `<Canvas>` in a React `<ErrorBoundary>` component that renders a CSS-only background (e.g., a radial gradient star field via `index.css`) when WebGL is unavailable. |
| 13 | 🟡 Warning | Visual Consistency | **Mixed animation systems**: The scroll indicator line uses Tailwind's `animate-pulse` (CSS keyframe, ~2 s loop) while every other animation in the section is GSAP-driven. This creates inconsistent timing, easing, and makes future `prefers-reduced-motion` support harder to apply uniformly. | `Hero.jsx:147` | Replace `animate-pulse` with a GSAP `gsap.to('.hero-scroll-line', { opacity: 0.2, repeat: -1, yoyo: true, duration: 1, ease: 'power1.inOut' })` call inside the existing entrance timeline, keeping all animation in one system. |
| 14 | 🟡 Warning | Visual Consistency | **Hardcoded `rgba(0,245,255,…)` in CTA hover handlers**: The `onMouseEnter`/`onMouseLeave` inline styles use literal RGBA values for the cyan color instead of reading `--color-accent-cyan`. Changing the design token will not update these hover colors. | `Hero.jsx:129–134` | Move hover styles to `index.css` using `.hero-cta:hover { background-color: rgba(from var(--color-accent-cyan) r g b / 0.1); }` or use a CSS variable for the rgba opacity variant (e.g., `--color-accent-cyan-10`). Remove the JS event handlers entirely. |
| 15 | 🟢 Suggestion | Perf | **Entrance animation takes ~2.2 s before CTA is visible**: Canvas fades 0→2 s; fades (including CTA) start at 1.2 s and finish at 2.0 s+stagger. Users cannot interact with "Ver proyectos" for over 2 seconds, increasing perceived wait time. | `hero.space.animations.js:28–43` | Tighten the timeline: start canvas fade at `duration: 1.2`, begin `.hero-fade` elements at `0.6` instead of `1.2`. The visual impact is preserved while reducing the interaction lock-in by ≈0.6 s. |
